# pnpm workspaces and strict node_modules

`pnpm` layouts can surface false positives when the dependency graph seen by source analysis does not line up with how tooling packages are actually consumed.

## Typical symptoms

- build tooling reported as unused
- missing dependency reports that look wrong in workspace setups
- confusion around packages that are available through workspace or hoisted layouts

## What to check first

- are the relevant files included in entry points?
- should the rule follow workspace packages?
- is the package only used indirectly by scripts or external tooling?
- do binary or module-usage hints belong in `pkgJsonFieldsWithBinaries`, `filesWithBinaries`, or `filesWithModules`?

## Example config

```jsonc
{
  "rules": [
    {
      "path": "packages/app",
      "followMonorepoPackages": true,
      "devEntryPoints": ["scripts/**", ".storybook/**/*.ts"],
      "unusedNodeModulesDetection": {
        "enabled": true,
        "pkgJsonFieldsWithBinaries": ["scripts"],
        "filesWithModules": [".storybook/main.ts"]
      },
      "missingNodeModulesDetection": {
        "enabled": true,
        "outputType": "groupByModuleFilesCount"
      }
    }
  ]
}
```

## Useful commands

```bash
rev-dep node-modules missing --group-by-module-files-count
rev-dep node-modules unused --entry-points src/main.ts
rev-dep resolve --module @acme/shared --follow-monorepo-packages
```

If the package is not statically imported from processed code, `rev-dep` may classify it differently from a script runner or bundler ecosystem.
