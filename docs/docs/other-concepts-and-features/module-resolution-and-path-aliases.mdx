# tsconfig paths and alias resolution

Rev-dep supports tsconfig path aliases, but alias setups can still create ambiguity in monorepos.

## Practical guidance

- prefer clear, non-overlapping alias targets
- avoid alias definitions that compete with workspace-package names unless the distinction is intentional
- verify whether the desired behavior is package-based resolution or tsconfig-path resolution

## Example tsconfig

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@ui/*": ["src/ui/*"],
      "@shared/*": ["../../packages/shared/src/*"]
    }
  }
}
```

Example rev-dep config:

```jsonc
{
  "rules": [
    {
      "path": "packages/app",
      "followMonorepoPackages": false,
      "unresolvedImportsDetection": {
        "enabled": true
      }
    }
  ]
}
```

## When results look surprising

Check:

- which `tsconfig.json` the command or rule is using
- whether a workspace package name and a tsconfig alias can both match the same request
- whether `followMonorepoPackages` is changing resolution expectations

## Practical limitation

If two resolution strategies can both plausibly target different files for the same request, the setup is inherently harder to analyze reliably. Favor alias schemes that make the intended target obvious.

## Useful commands

```bash
rev-dep unresolved --tsconfig-json tsconfig.json
rev-dep resolve --file src/shared/button.ts --tsconfig-json tsconfig.json
```

# package.json imports and exports maps

Rev-dep supports `package.json` import and export map resolution, including configurable condition names.

## When this matters

- you use Node-style import maps instead of tsconfig aliases
- workspace packages expose subpaths through `exports`
- resolution behavior changes across `default`, `node`, or custom conditions

## Related settings

- top-level `conditionNames`
- rule-level `followMonorepoPackages`

Example:

```jsonc
{
  "configVersion": "1.7",
  "conditionNames": ["node", "default"],
  "rules": [
    {
      "path": "packages/app",
      "followMonorepoPackages": true,
      "unresolvedImportsDetection": {
        "enabled": true
      }
    }
  ]
}
```

If map-based resolution still looks wrong, compare the expected runtime condition set with the condition names passed to `rev-dep`.

## Why condition names matter

The same package can resolve to different targets under conditions like:

- `development`
- `production`
- `browser`
- `node`
- `default`

If your project relies on a non-default condition path, document that explicitly in config instead of assuming the default resolution matches your runtime.

## Example package.json shapes

```json
{
  "imports": {
    "#shared/*": "./src/shared/*.ts"
  },
  "exports": {
    ".": {
      "default": "./src/main.ts",
      "node": "./src/node.ts"
    }
  }
}
```

And a useful exploratory command:

```bash
rev-dep unresolved --condition-names node,default
```
